# Notion Auto Lock Worker

Unofficial [Notion Worker](https://developers.notion.com/workers/get-started/overview) that automatically locks stale pages in configured Notion root pages and data sources.

> Notion Workers is currently in beta. APIs, CLI commands, pricing, templates, and hosting behavior may change.

This example project was generated by `create-notion-auto-lock-worker`. It is not made by, endorsed by, or affiliated with Notion.

## Prerequisites

- Node.js 22 or later
- npm 10.9.2 or later
- [Notion CLI](https://developers.notion.com/reference/cli) (`ntn`)
- A Notion Business or Enterprise workspace with [Workers](https://developers.notion.com/workers/get-started/overview) enabled
- A Notion connection with [`Read content` and `Update content` capabilities](https://developers.notion.com/reference/capabilities)

Install the Notion CLI:

```bash
curl -fsSL https://ntn.dev | bash
ntn --version
```

## Setup

Install dependencies:

```bash
npm install
npm run check
```

Log in to Notion:

```bash
ntn login
```

If your Notion account has multiple workspaces, you can keep the CLI target workspace in an ignored local file:

```bash
printf "NOTION_WORKSPACE_ID=your_workspace_id_here\n" > .env.cli
set -a
source .env.cli
set +a
ntn debug
```

Create a Notion connection in the developer portal, then grant it the required [capabilities](https://developers.notion.com/reference/capabilities):

- Read content
- Update content

If you use an [internal connection](https://developers.notion.com/guides/get-started/internal-connections), open each root page or database in Notion and add the connection from the page or database `Connections` menu. A connection can read and update shared pages and their children. [Personal access tokens](https://developers.notion.com/guides/get-started/personal-access-tokens) use your own Notion access and do not need that manual share step.

## Configuration

Store runtime values as [Worker secrets](https://developers.notion.com/workers/guides/secrets):

```bash
ntn workers env set AUTO_LOCK_API_TOKEN=ntn_...
ntn workers env set AUTO_LOCK_ROOT_PAGE_IDS=...
ntn workers env set AUTO_LOCK_DATA_SOURCE_IDS=
ntn workers env set WORKER_SCHEDULE=3h
ntn workers env set LOCK_AFTER_MINUTES=60
ntn workers env set DRY_RUN=true
ntn workers env set LOCK_ROOT_PAGES=false
ntn workers env set MAX_CRAWL_DEPTH=10
ntn workers env set MAX_CRAWL_PAGES=1000
```

Or keep them in an ignored local env file and push them together:

```bash
cp .env.example .env.production
ntn workers env push --file .env.production
```

Never commit `.env`, `.env.cli`, or `.env.production`.

| Name | Required | Default | Description |
| --- | --- | --- | --- |
| `AUTO_LOCK_API_TOKEN` | yes | none | Notion API token |
| `AUTO_LOCK_ROOT_PAGE_IDS` | conditional | none | Comma-separated root page IDs to crawl recursively |
| `AUTO_LOCK_DATA_SOURCE_IDS` | conditional | none | Comma-separated data source IDs to query directly |
| `WORKER_SCHEDULE` | no | `3h` | Scheduled sync interval, from `5m` to `7d` |
| `LOCK_AFTER_MINUTES` | no | `60` | Minutes after last edit before a page becomes eligible |
| `DRY_RUN` | no | `true` | Count eligible pages without locking them |
| `LOCK_ROOT_PAGES` | no | `false` | Lock configured root pages themselves; by default root pages are crawl anchors only |
| `MAX_CRAWL_DEPTH` | no | `10` | Maximum nested page/block/database depth to crawl |
| `MAX_CRAWL_PAGES` | no | `1000` | Maximum pages to crawl in one run |

Set at least one of `AUTO_LOCK_ROOT_PAGE_IDS` or `AUTO_LOCK_DATA_SOURCE_IDS`.

`AUTO_LOCK_DATA_SOURCE_ID` is still accepted for compatibility, but new projects should use `AUTO_LOCK_DATA_SOURCE_IDS`.

Scaffold-time values are compiled into `src/index.ts`:

| Name | Default | Description |
| --- | --- | --- |
| `AUDIT_DATABASE_TITLE` | `Auto Lock Runs` | Managed audit database title |
| `NOTION_API_VERSION` | `2026-03-11` | Pinned Notion API version |

Change scaffold-time values in `src/index.ts`, then redeploy.

## Schedule And Cost

`WORKER_SCHEDULE` defaults to `3h`. Set it as a Worker secret when you want to change the execution interval after scaffolding, then redeploy so Notion refreshes the sync capability. Notion's [Workers syncs guide](https://developers.notion.com/workers/guides/syncs#set-a-schedule) documents interval schedules such as `5m`, `15m`, `1h`, and `1d`, with a minimum of `5m` and a maximum of `7d`.

As of 2026-06-23, Notion's [Workers pricing guide](https://www.notion.com/help/understand-pricing-for-workers) says Workers are free to try during beta on Business and Enterprise plans, and are expected to require Notion credits starting August 11, 2026. The same guide says each scheduled sync execution counts as one Worker run, Workers typically cost about `$0.0023` per run, and actual usage may vary based on how much work a Worker does.

Approximate run and cost examples, assuming `$0.0023` per run:

| Schedule | Runs / month | Approx cost / month |
| --- | ---: | ---: |
| `1d` | 30 | `$0.07` |
| `1h` | 720 | `$1.66` |
| `30m` | 1,440 | `$3.31` |
| `15m` | 2,880 | `$6.62` |
| `5m` | 8,640 | `$19.87` |

The `1d`, `1h`, and `15m` examples align with Notion's published examples. `30m` and `5m` are simple estimates using the same per-run rate.

With `LOCK_AFTER_MINUTES=60` and `WORKER_SCHEDULE=3h`, pages are locked on the next scheduled run after they become at least 1 hour old. In practice, that means roughly 1 to 4 hours after the last edit.

Root page crawling can issue many Notion API requests because it walks child pages, child databases, data sources, and database pages. Keep `DRY_RUN=true` at first and review `MAX_CRAWL_DEPTH` and `MAX_CRAWL_PAGES` before enabling live locking.

## Dry Run

Dry run is enabled by default. Deploy once with `DRY_RUN=true`, then inspect logs and the `Auto Lock Runs` audit database.

```bash
npm run deploy
```

Trigger the sync manually:

```bash
npm run dry-run
```

When the output looks right, switch to live locking:

```bash
ntn workers env set DRY_RUN=false
npm run deploy
```

## Behavior

Each run:

1. Calculates a cutoff time from `LOCK_AFTER_MINUTES`.
2. Crawls configured root pages recursively. Root pages are crawl anchors by default.
3. Follows child pages, child databases, and data sources found under those root pages.
4. Queries configured data sources directly.
5. Retrieves each candidate page again immediately before locking.
6. Locks the page only if it is still older than the cutoff and still unlocked.
7. Writes one summary row to the managed audit database.

The lock operation sets [`is_locked: true`](https://developers.notion.com/reference/patch-page#locking-and-unlocking-a-page). This locks editing in the Notion app UI, but Notion API updates can still modify the page.

Configured root pages are not locked unless `LOCK_ROOT_PAGES=true`.

Linked databases and linked data sources are not expanded by the Notion API. Share the original database or data source with the connection, or include it in `AUTO_LOCK_DATA_SOURCE_IDS`.

## Logs And Audit

Logs include only operational counts and shortened IDs:

- run id
- dry run status
- scope counts
- checked page count
- eligible page count
- locked page count
- skipped page count
- error count
- crawled page count
- crawled block count
- crawled database count
- crawled data source count
- crawl limit status
- elapsed milliseconds
- error code

Logs and audit records do not include page titles, page content, property values, user names, emails, full page URLs, or tokens.

## Troubleshooting

Use:

```bash
npm run sync:status
```

Common issues:

- `configuration_error`: a required Worker secret is missing or invalid.
- `unauthorized`: `AUTO_LOCK_API_TOKEN` is invalid.
- `restricted_resource`: the connection lacks access or the required capability.
- `object_not_found`: a root page, database, or data source is missing or not shared with the connection.
- `crawlLimitReached=true`: increase `MAX_CRAWL_PAGES`, lower the scope, or split the scope across multiple Workers.
- High cost estimate: increase `WORKER_SCHEDULE`, for example from `15m` to `1h`.

## Security

- Keep `AUTO_LOCK_API_TOKEN` in Worker secrets or local `.env` only.
- Do not commit `.env`.
- Use the minimum Notion capabilities: [`Read content` and `Update content`](https://developers.notion.com/reference/capabilities).
- Prefer an [internal connection](https://developers.notion.com/guides/get-started/internal-connections) token when you want narrower access.
- Share only the root pages, databases, or data sources that this Worker should be allowed to lock.
- Review `WORKER_SCHEDULE` before enabling live locking so Worker run volume is predictable.
